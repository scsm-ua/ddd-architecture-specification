# Концепция архитектуры образовательного приложения в рамках миссии проповеди.

## Ключевая идея

- **текстовоые документы - главный ресурс**, который передается следующему поколению (сюда можно отнести картинки и видео)
- **код - вторичный ресурс**, который можно периодично перегенерировать под новые технологии для отображения старого контента
- открытая децентрализированная инфраструктура, с прозрачной спецификацией, которую смог бы поддерживать ИИ

## Слои данных

1) текстовые данные - книги, статьи, лекции, разговоры
2) видео, аудио, фото - медия контент (возможно связан с п.1)
3) графические изображения - медия для п.1, п.4
4) лейаут и логика приложения для просмотра разных типов контента - UX
5) графический дизайн приложения - UI
6) спецификация логики приложения - генерации кода, который рендерит п.1, п.2, п.3, согласно логике п.4, п.5
7) спецификация архитектуры разворачиваня приложения, данных и бекапов - в т.ч. аутентификация юзеров, админы, децентрализация
8) спецификация архитектуры хранения каждого типа данных п.1-п.7 - обеспечение сохранности архива, и возможность обновления каждого слоя архитектуры, независимо от других.

## Ключевые тезисы

1) Наибольшую ценность представляет исходный контент из п.1 и п.2
2) Дизайн (п.3-5) - ценный ресурс, так как красота и удобство подачи - важная составляющая представления контента.
3) Наименьшую ценность на сегодня представляет код приложения (генерируется по п.6), так как с ИИ генерацией его сделать быстрее и дешевле всего остального.
4) Один из хороших методов хранения текстового контента п.1 - **децентрализированные git репозитории и markdown формат** (см.далее).
5) Вопрос хранения медия контента, важны бекапы в разных зонах:
	- s3 storage (или ftp) - просто зеркально копировать, синхронизировать, хранить домен разных зеркал в конфиге и переключать для относительных урлов
	- бесплатные медия сервисы (youtube, archive.org)
	- реестр для учета храненной медии (одно место, где есть ссылки на места хранения контента)
6) Схема данных для маркдаун документов
	- каждый документ имеет свою схему, зависмо от типа контента (книга, глава, стих, песня, лекция)
	- схема определяет структуру и допустимое форматирование под дизайн
	- валидация документа по схеме для юзера, чтобы он не нарушил дизайн

## Сложные моменты

1) Миграция схем
	- поддержка обратной совместимости новыми апами старых данных, и старыми новых
2) Администрирование бекапов
3) Администрирование деплоймента
4) Администрирование авторизации
5) Редактирование контента - если есть схема и источник данных, то можно сделать одно место для редактирования, а не в каждом приложении отдельно.
6) Tracability - нужна возможность для остлеживания всех связей: бекапы-контент-приложения, чтобы изменения в одной сущности имели предсказуемые последствия для завязок.

## Децентрализированные git репозитории и данные в markdown формате

1) Plain text
	- можно редактировать и просматривать где угодно
	- можно кастомизировать, главное не сильно усложнять
	- простой changes control
	- удобен для ИИ
	- самый малый размер файлов
2) Git
	- децентрализация
	- source control
	- authorization contol
	- бесплатный бекап на сервисах типа github, bitbucket
3) Универсальнйы источник для любых приложений
	- легко парсить
	- легко импортировать из git
	- можно генерировать книгу для верстки

Таким образом, markdown+git может быть единым редактируемым источником для:

1) Сайтов, приложений
2) Генерации книг
3) Баз знаний для ИИ
